What the hell it is

Oh Vince, should we really spend time knowing a data structure that seems like an array and may possible not result in a significant change in time complexity of queries and insertions? Isn’t it just another computational scam that only serves as differential trick for interviews between innocent college students and aged senior programmers?

A linked list is sort of like an array, but quite different in terms of location of memory. TYPICALLY arrays usually take up consecutive blocks of data on the RAM - a characteristic that signifcantly boosts our speed to access an element at a specific index. Linked lists are different however, as the elements of a linked list is stored not contigously on the memory locations, rather separated in multiple locatons.

This defines the major characteristic of linked list and further leads to many differences with linked lists and arrays.

  1. A linked list is dynamic in size, but an array is fixed in size.
  2. A linked list uses more memory as both the address of the next node and the value of current node have to be stored. An array doesn’t have this concern, as it’s typically storing one value only.
  3. A linked list cannot directly access an element at a specific index for O(1). It needs to traverse the linked list to find its locaton, hence giving it a time complexity of O(n). An array, of course, accesses element in a O(1) complexity.
  4. A linked list process insertions and deletions much quicker, as only the address of node needs to be altered. An array would have to do this in a timely manner.

Logic:

  4. Iterate the linked list. temp is the next node we’re processing. Firstly we invert curr’s next so it points to the former node instead of the latter.
